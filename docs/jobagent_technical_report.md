# JobAgent 学生成长诊断与岗位匹配系统技术报告

生成日期：2026-06-09  
项目目录：`/Users/sunnychen/Dev/JobAgent`  
报告对象：学生能力画像、岗位匹配、成长路径规划一体化系统

## 1. 摘要

JobAgent 是一套面向大学生求职准备和高校就业指导场景的 Agentic 诊断系统。系统从简历、成绩单和证书等材料出发，完成材料结构化、能力评分、岗位推荐、差距分析和成长路径规划。当前版本的核心边界是：Go 后端作为统一网关和异步任务编排层，Legato 负责简历/成绩单快速结构化，Presto 负责多 Agent 推理与结构化输出，前端通过 SSE 展示从上传到画像、匹配、路径、导出的实时进度。

系统已经形成三类核心输出：能力画像、路径规划和匹配结果。能力画像支持 JSON 和 Excel 导出，包含基础信息、教育背景、证据评分、六维雷达数据和 TOP5 匹配岗位；路径规划由 Path Planning Agent Team 生成，包含阶段目标、周任务、资源链接、达标标准和交付物，并在前端支持 PDF 打印、后端支持 Word 样式导出；匹配结果以可视化报表展示岗位匹配度、个人/岗位目标雷达、差距明细和推荐理由。

需要说明的是，用户硬性指标中提到“四维分值”，当前代码实现已经扩展为更细的六维能力口径：逻辑、语言、专业、领导、抗压、成长。报告建议在交付层保留六维主口径，同时提供四维兼容映射：学业基础、专业实践、组织协作、成长韧性。这样既符合现有算法，又能满足外部验收表述。

## 2. 系统总体架构

### 2.1 分层架构

系统采用“前端体验层 + Go 诊断网关 + Legato 结构化引擎 + Presto Agent 运行时”的四层架构。

| 层级 | 主要模块 | 当前职责 | 关键文件 |
|---|---|---|---|
| 前端体验层 | `frontend/index.html`、`frontend/app.js`、`frontend/styles.css` | 上传材料、展示异步进度、渲染能力雷达、岗位匹配报表、路径规划和导出按钮 | `frontend/README.md` |
| Go 诊断网关 | `backend/main.go`、`backend/async.go` | 静态资源托管、上传接收、异步 JobStore、SSE、导出、调用 Legato 与 Presto Agent Team | `backend/README.md` |
| Legato 结构化引擎 | `Agents/legato/legato/*`、`Agents/legato/workflows/resume/*` | PDF 快速文本层抽取、MarkItDown 回退、清洗、简历/成绩单 schema 格式化、resume workflow 阶段执行 | `Agents/legato/docs/ARCHITECTURE.md` |
| Presto Agent 运行时 | `Agents/presto/internal/*` | session/run API、SSE run events、OpenAI-compatible provider、并发 workflow、重试和结构化输出校验 | `Agents/presto/README.md` |

### 2.2 端到端数据流

1. 学生在前端上传简历，成绩单和其他材料为可选。
2. Go 后端创建异步诊断任务，返回 `job_id` 和 SSE 地址。
3. 后端并发启动 Legato resume workflow 的 `profile`、`certifications_awards`、`experience_hybrid` 阶段，同时尝试处理成绩单。
4. 简历解析成功后，前端展示基础画像和证据卡片；成绩单失败会作为可选材料警告，不阻断诊断。
5. 用户或前端触发 Benchmark 后，后端并发调用 Item Benchmark 和 Major Baseline：
   - Item Benchmark 为每条证据生成六维分布和 `impact_factor`。
   - Major Baseline 为专业、学历和学业信号生成六维 academic prior。
6. Go 后端统一聚合 `radar_data` 和 `radar_series`，前端不再自行计算画像分值。
7. Benchmark 和 Major Baseline 完成后，后端启动 Legato Job Matching Team：
   - Adaptive Planner 先判断复杂度并派生 3 到 6 个多视角 Agent。
   - 多视角 Agent 并发分析能力、证据、学历门槛、岗位族和风险。
   - Synthesis Arbiter 汇总输出岗位匹配 JSON，后端校验雷达一致性。
8. Job Matching 完成后，Path Planning Team 基于首选岗位、差距明细和发展动作生成成长路径。
9. 前端解锁结构化输出，支持画像 JSON/Excel、匹配 JSON/PDF、路径 JSON/Word/PDF。

### 2.3 当前实现边界

当前系统已经形成正式诊断流程：简历 Legato 解析是必需步骤，岗位推荐和路径规划由 Presto Agent Team 生成。系统以真实上传材料、结构化证据和 Agent 输出作为前端展示与导出的统一来源。

仍需生产化的部分包括：对象存储、认证授权、隐私脱敏、真实岗位库或招聘平台数据源、扫描版成绩单稳定 OCR、后端模板化 PDF 服务、多实例下任务状态持久化。这些待补项已在 `backend/REQUIREMENTS.md` 和 `backendRequirements()` 中列出。

## 3. 核心结构化输出

### 3.1 能力画像

能力画像对应 Go 结构体 `AbilityProfile`，核心字段包括：

| 字段 | 含义 |
|---|---|
| `basic_info` | 姓名、性别、出生年份、学校、专业、学历、毕业年份、首选岗位、简历/成绩单状态 |
| `education` | 学校、院系、专业、学历、985/211/双一流、软科排名、独立学院标记 |
| `radar_data` | 六维能力主分值，用于最终学生雷达 |
| `radar_series` | 综合、校内、校外三条雷达序列 |
| `awards` | 奖项证书证据、level、impact、六维分布、评分说明 |
| `experiences` | 实习、项目、校园经历证据、level、impact、六维分布、评分说明 |
| `major_baseline` | 专业族群、基础分、六维 academic prior、置信度 |
| `top5_matching_jobs` | TOP5 推荐岗位、匹配度、推荐理由、下一步证据 |

导出方式：

| 格式 | 当前入口 | 内容 |
|---|---|---|
| JSON | `GET /api/export/ability-profile.json`，前端 `导出 JSON` | 完整 `AbilityProfile` |
| Excel | `GET /api/export/ability-profile.xlsx`，前端 `导出 Excel` | 基础信息、教育经历、专业基线、六维指标、奖项、经历、TOP5 岗位 |
| PDF | 前端 `printSectionAsPDF("profile")` | 通过浏览器打印画像模块生成 |

### 3.2 路径规划

路径规划对应 `PathPlan`，由 Path Planning Agent Team 输出。每个阶段包含：

| 字段 | 要求 |
|---|---|
| `stage` | 阶段名称，例如“第 1 阶段，0 到 30 天” |
| `goal` | 阶段目标，必须绑定首选岗位和主要短板 |
| `weeks` | 3 到 5 个周任务，每个任务包含周次、任务、达标指标、优先级 |
| `resources` | 稳定可访问资源链接，后端过滤非 `http` URL |
| `standards` | 至少 2 条可验证达标标准 |
| `deliverable` | 阶段交付物，例如作品集链接、测试报告、简历定稿 |

导出方式：

| 格式 | 当前入口 | 状态 |
|---|---|---|
| Word | `GET /api/export/path-plan.doc`，前端 `导出 Word` | 后端返回 Word 可打开的路径规划文档，诊断路径由 Agent Team 填充 |
| PDF | 前端 `printSectionAsPDF("path")` | 使用浏览器打印样式生成 |
| JSON | 前端 `导出 JSON` | 完整 `PathPlan` |

### 3.3 匹配结果

匹配结果对应 `MatchingResult`，核心内容包括：

| 字段 | 含义 |
|---|---|
| `target_role` | 系统自动推荐的首选岗位 |
| `overall_match` | 总体匹配度，0-100 |
| `match_level` | 匹配等级，例如强匹配、高潜力匹配、可迁移匹配、需补证据 |
| `selected_job` | 首选岗位对象，包含能力匹配、经历匹配、学历门槛、推荐理由和下一步证据 |
| `student_radar` | 学生六维能力 |
| `target_radar` | 岗位目标六维能力 |
| `report_sections` | 逐维差距报表，包含学生分、岗位要求、差距、状态和说明 |
| `gap_details` | 具体短板、当前状态、预期状态和行动建议 |
| `development_actions` | 按校内/校外、优先级结构化的发展动作 |
| `recommendations` | 投递和补证据建议 |

前端可视化包括：匹配度、岗位队列、个人画像与岗位目标雷达、平均差、最大短板、优势项、逐维差距卡、差距明细和推荐理由。匹配报表支持 JSON 下载和 PDF 打印。

## 4. 评分标准

### 4.1 证据 Level 评分

经历 `experience.level` 使用 0-10 分，评分原则是保守评价简历文本中的证据，而不是只看品牌、奖项名或头衔。评分由六个因素组成：

| 维度 | 分值 | 说明 |
|---|---:|---|
| 与目标岗位相关性 | 0-2.0 | 是否直接对应目标岗位或技术/业务域 |
| 技术/专业深度 | 0-2.0 | 是否体现工程、分析、研究、产品、数据、算法、测试或领域深度 |
| 个人 ownership | 0-2.0 | 是否独立负责、主导、设计、分析或解决具体问题 |
| 结果/影响 | 0-1.5 | 是否有获奖、排名、上线、可衡量结果或明确改进 |
| 范围/选择性 | 0-1.5 | 国家级、省级、区域、校级、知名组织等选择性 |
| 证据质量 | 0-1.0 | 是否有具体动作和结果，是否避免只有标题 |

分段解释：

| Level | 解释 |
|---:|---|
| 9-10 | 极强证据，通常需要高选择性、核心 ownership 和明确结果 |
| 7-8 | 强证据，技术/研究/竞赛贡献清晰，有实际产出 |
| 5-6 | 有效证据，有相关项目或校园/实习贡献，但深度或影响有限 |
| 3-4 | 低信号，例行标注、审核、简单参与或描述弱 |
| 1-2 | 弱证据，多为泛荣誉、基础证书或 title-only |
| 0 | 无效或不可用 |

基础证书、泛荣誉、奖学金、缺少贡献描述的奖项会被限制上限；外包式标注或简单审核任务没有技术 ownership 时上限为 4；全国级竞赛如果只列奖项、不写个人贡献，也不会自动进入高分段。

### 4.2 Item Benchmark 与六维能力

Item Benchmark 对奖项和经历证据输出两个结果：

1. `impact_factor`：0-10，衡量该证据对能力判断的影响强度。
2. `scores`：六维权重向量，顺序固定为逻辑、语言、专业、领导、抗压、成长，取值 0.0-1.0，合计为 1.0。

六维定义如下：

| 六维 | 定义 |
|---|---|
| 逻辑 | 数学、科学、分析、建模和问题拆解能力 |
| 语言 | 写作、沟通、表达、陈述和人文表达能力 |
| 专业 | 与学生专业或技术领域直接相关的能力 |
| 领导 | ownership、组织协调、团队影响和主导能力 |
| 抗压 | 约束、压力、难度、坚持和交付能力 |
| 成长 | 学习潜力、主动性、迭代和探索能力 |

硬性指标中的四维分值可以由六维映射得到：

| 四维兼容口径 | 六维来源 | 用途 |
|---|---|---|
| 学业基础 | 逻辑、语言、专业基线 | 面向学校侧汇报学生学业与表达基础 |
| 专业实践 | 专业、逻辑、项目/实习证据 | 面向岗位专业胜任力 |
| 组织协作 | 领导、语言 | 面向团队、沟通、活动和 ownership |
| 成长韧性 | 成长、抗压 | 面向可塑性、执行力和持续投入 |

### 4.3 雷达聚合公式

Go 后端统一聚合最终雷达，前端只读取 `radar_data` 和 `radar_series`。核心公式如下：

1. 证据强度：

```text
evidence_strength = 0.4 * level / 10 + 0.6 * impact_factor / 10
```

2. 单维贡献：

```text
dimension_contribution = six_dim_score * evidence_strength
dimension_contribution <= 0.96
```

3. 低置信证据使用 bucket 双重上限，包括基础证书/低影响奖项、校内泛奖项、泛校园角色、无标题项目、低/中/高/极高影响证据。弱证据不能靠数量堆出强能力。
4. 多条证据使用强递减收益。证据分达到约 65 后边际收益明显降低，达到约 70 后边际收益约为 4%。
5. 最终分不是证据和基线简单平均，而是：

```text
final_ability = school_tier_prior + evidence_lift
```

同时受学校层级 ceiling 限制。学校层级主要影响逻辑、专业、抗压和成长，不会单独抬高领导维度。

### 4.4 专业与学校基线

Major Baseline 为学生专业和学业记录生成六维 academic prior。专业族群包括文科类、理科类、工科类、商科类、医农类、艺术体育类、交叉类、未知。成绩不是直接能力分：均分 80 左右只代表普通学业完成，约映射到 50 的能力 prior；85 约 58；90 约 66；95 约 74。

学校层级 ceiling 当前实现如下：

| Tier | 默认 Base | 无高影响证据上限 | 有高影响证据上限 | 极强证据上限 |
|---|---:|---:|---:|---:|
| T0：985 或软科 <= 50 | 68 | 78 | 92 | 94 |
| T1：211/双一流或软科 <= 150 | 62 | 72 | 86 | 88 |
| T2：非双一流软科 151-250 | 52 | 62 | 76 | 78 |
| T3：普通本科/未知/更低排名 | 46 | 56 | 68 | 72 |
| T4A：独立学院背景但后续有更强学历 | 45 | 54 | 66 | 70 |
| T4：独立/民办/原三本 | 38 | 52 | 60 | 62 |
| T5：专科 | 34 | 48 | 56 | 60 |

### 4.5 岗位匹配评分

岗位匹配不是单一学历筛选，而是三类信号综合：

1. 六维能力拟合：以学生 `radar_context.overall` 和岗位 `requirement_radar` 对比为第一排序信号。
2. 经历相关性和证据质量：实习、项目、竞赛、作品集等能否证明岗位关键能力。
3. 学历门槛与风险：学历是 gate 和风险约束，不单独决定排序；高 impact 且高度相关的项目可以部分突破门槛，但必须暴露风险和下一步证明。

匹配等级由总分归类：

| 匹配度 | 等级 |
|---:|---|
| >= 85 | 强匹配 |
| 75-84 | 高潜力匹配 |
| 65-74 | 可迁移匹配 |
| < 65 | 需补证据 |

## 5. Agentic 架构

### 5.1 Legato Resume Workflow

Legato 的基本职责是将材料转为结构化 JSON。PDF 简历优先走 `pypdfium2` 文本层快速路径，文本不足时回退 MarkItDown；扫描材料可显式启用 PaddleOCR，但当前 macOS CPU 实测较慢，仍属于实验能力。

Resume workflow 当前并发运行：

| Agent/阶段 | 输出 |
|---|---|
| `profile` | identity、education、school tags、degree level |
| `certifications_awards` | 奖项与证书 |
| `experience_hybrid` | 项目、实习、校园角色，整体替换最终经历列表 |
| `transcript` | 可选成绩单解析，失败不阻断简历诊断 |

### 5.2 Benchmark 双通道

Benchmark 阶段同时运行：

1. Item Benchmark：证据按批并发，默认最多 5 个请求，每批默认 2 个 worker，可通过环境变量调整。
2. Major Baseline：根据专业、学校层级、学历和成绩单线索输出六维 academic prior。

任意 Item Benchmark 批次返回后，后端会通过 SSE 更新对应证据卡片；Major Baseline 返回后补齐专业六维基线。二者全部完成后才允许 Job Matching。

### 5.3 Job Matching Agent Team

Job Matching 是当前系统最完整的 Agentic 模块。流程如下：

1. Adaptive Planner 读取学生画像、证据数量、专业基线、学历门槛、简历全文上下文和复杂度信号。
2. Planner 输出受限 Agent plan，后端校验 3 到 6 个 Agent，最多 3 个并发 run。
3. 多视角 Agent 并发分析，必须覆盖 ability fit、evidence quality、education threshold、role family mapping；复杂案例可增加 growth potential、counterfactual risk、market 等视角。
4. Synthesis Arbiter 汇总所有视角，输出严格 JSON：首选岗位、TOP5、匹配度、目标雷达、差距明细、推荐理由、发展动作。
5. 后端校验输出：`student_radar` 必须与后端权威画像一致，`target_radar` 必须与首选岗位 requirement radar 一致；不合格会自动触发严格 JSON retry。

这套模式不是静态 prompt，而是“先规划团队，再并发推理，再仲裁校验”的 Agentic 工作流。

### 5.4 Path Planning Agent Team

路径规划只在岗位匹配完成后启动。流程为：

1. Path Planner 基于 `selected_job`、`report_sections`、`gap_details` 和 `development_actions` 判断规划复杂度。
2. Planner 派生 3 到 5 个路径规划 Agent，必需视角包括 `stage_goal`、`weekly_tasks`、`acceptance`；证据弱或风险高时增加 `evidence_delivery`、`schedule_risk`。
3. 多 Agent 并发输出阶段建议、周任务、资源、guardrails。
4. Path Synthesis Arbiter 合成为 2 到 4 个阶段，每阶段包含目标、3 到 5 个周任务、资源、达标标准和交付物。
5. 后端规范化输出，过滤无效资源链接和不完整阶段。

### 5.5 Presto 运行时价值

Presto 提供 session/run API、异步 run、SSE event replay、token stream、OpenAI-compatible provider、Echo provider、可选响应缓存、工具重试和结构化 contract retry。JobAgent 利用这些能力将每个 Agent 的状态流转发给前端，前端 chat 中以结构化状态卡展示 Planner、多视角 Agent 和 Arbiter 的流式进度。

## 7. 创新之处

### 7.1 多维度能力量化

系统不是把简历简单归纳为“强/弱”，而是将每条证据拆成 level、impact_factor 和六维分布，再通过学校/专业 prior、证据递减收益和证据 bucket 上限生成最终画像。这使得基础证书、泛荣誉和强项目不会被同等看待，能减少“证书堆叠型”误判。

### 7.2 三方数据闭环

系统形成“学生材料 - 岗位要求 - 成长任务”的闭环：学生材料生成能力画像，岗位匹配生成目标雷达和差距，路径规划再把差距转成周任务、资源和达标标准。后续接入真实岗位库和课程库后，可扩展为“学生 - 学校课程 - 企业岗位”的三方数据闭环。

### 7.3 双向匹配算法

岗位推荐不是只从学生简历找关键词，也不是只按岗位学历门槛筛人，而是将学生六维画像与岗位目标雷达双向对齐。系统同时输出匹配度、差距、风险、下一步证据，使推荐岗位既可解释又可行动。

### 7.4 AI 个性化路径

路径规划不是固定模板。Path Planner 会基于首选岗位、最大短板、证据缺口和发展动作动态派生 Agent，并要求每个阶段产生可验证指标和交付物。学生获得的是面向目标岗位的 60-120 天行动计划，而不是泛泛学习建议。

### 7.5 课程 - 岗位智能映射

当前 `backend/REQUIREMENTS.md` 已将“成绩单解析与课程能力映射”列为 P0，现有 Major Baseline 已具备专业族群和成绩线索对六维 prior 的建模能力。下一步接入课程类别、核心课程成绩和课程项目后，可以将课程证据映射到岗位能力维度，形成课程到岗位的智能映射。

## 8. 当前风险与改进建议

| 风险 | 当前状态 | 建议 |
|---|---|---|
| 扫描版成绩单 OCR 不稳定 | Chenxi 成绩单离线模式未抽取课程 | 部署稳定 OCR 服务，优先处理成绩单表格结构 |
| 真实岗位数据源缺失 | 当前由 Agent 根据画像推断岗位 | 接入岗位库/JD 数据源，支持地区、学历、技能过滤 |

| 多实例恢复 | 当前 JobStore 是内存态 | 将任务、事件、文件元数据持久化 |
| 隐私与权限 | 尚未实现认证授权和脱敏日志 | 增加登录、访问控制、审计、删除和保留周期 |

## 9. 结论

JobAgent 当前已经具备从学生材料到能力画像、岗位匹配、成长路径和结构化导出的完整技术骨架。它的关键价值不在于单次简历总结，而在于将材料解析、证据评分、岗位目标和成长任务连接为可解释、可追踪、可导出的 Agentic 工作流。

从工程成熟度看，Legato 的文本层简历解析和 Go 后端异步编排已经成型；Job Matching Team 和 Path Planning Team 已形成动态 Agent 编排、并发推理、严格 JSON 重试和后端校验机制；前端已经能用 SSE 展示诊断进度和可视化差距。下一阶段应优先补齐 OCR、岗位库、导出服务、持久化和数据安全，以把当前技术原型提升为可稳定交付的生产系统。
